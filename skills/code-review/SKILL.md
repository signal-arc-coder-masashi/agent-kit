---

## 概要

シニアソフトウェアアーキテクトとしてコードベース全体を監査する。
4つの評価軸（品質・セキュリティ・設計・コスト）ごとに具体的な問題を特定し、
**ファイルパスと行番号を明示した上で、改善ポイントをコード例つきで提示する**。

---

## Step 1: プロジェクト検出

まず構造を把握する。

```bash
# プロジェクトルートを確認
ls -la
find . -maxdepth 2 -name "package.json" -o -name "requirements.txt" \
  -o -name "go.mod" -o -name "Cargo.toml" -o -name "Gemfile" \
  -o -name "pom.xml" -o -name "build.gradle" | head -20
```

確認する情報：
- **言語・フレームワーク**（package.json の dependencies、import文のパターン等）
- **エントリーポイント**（main.py / index.ts / app.go 等）
- **DB/API接続**（ORM、クエリ、外部API呼び出し）
- **設定ファイル**（.env.example、config/、docker-compose.yml 等）

---

## Step 2: コードスキャン（戦略的に読む）

全ファイルを読む必要はない。重要箇所に集中する。

### 必ず確認する場所
1. エントリーポイントと主要モジュール
2. データアクセス層（DB クエリ、API 呼び出し）
3. 認証・認可ロジック
4. 環境変数・設定の扱い
5. 依存パッケージ一覧（バージョン含む）

### Grep で探すパターン

```bash
# シークレット漏洩の疑い
grep -rn "password\s*=\s*['\"]" . --include="*.py" --include="*.ts" --include="*.js"
grep -rn "api_key\s*=\|API_KEY\s*=" . --include="*.py" --include="*.ts" --include="*.js"

# 外部API呼び出し
grep -rn "fetch(\|axios\.\|anthropic\.\|openai\.\|requests\.get\|requests\.post" . --include="*.py" --include="*.ts" --include="*.js"

# N+1 クエリの疑い（ループ内のクエリ）
grep -rn "for.*in\|\.forEach\|\.map(" . --include="*.py" --include="*.ts" | head -30

# SQL クエリ
grep -rn "SELECT\|INSERT\|UPDATE\|DELETE\|execute(" . --include="*.py" --include="*.ts" --include="*.js"

# エラーハンドリングの欠如
grep -rn "except:\|catch(e)\|catch(err)" . --include="*.py" --include="*.ts" --include="*.js"
```

## Step 3: 4軸評価

各問題は以下の形式で記録する：
- 📍 ファイルパス（行番号）
- ❗ 問題の内容
- 💡 改善ポイント（コード例あり）
- 🔴重大 / 🟡中程度 / 🟢軽微

---

### 品質 (Quality)

| チェック項目 | 何を見るか |
|---|---|
| 重複コード | 同じロジックが複数箇所に存在しないか |
| 関数の複雑さ | 50行超・分岐5つ超の関数はリファクタ対象 |
| エラーハンドリング | 未処理の Promise、bare `except:`、null チェック漏れ |
| テスト | テストファイルの有無、重要パスのカバレッジ |
| 可読性 | マジックナンバー、不明瞭な変数名、型定義の欠如 |

**改善ポイントの書き方例：**
```
❌ 問題のあるコード
✅ 改善後のコード
📝 理由：なぜこの変更が効くか
```

---

### セキュリティ (Security)

| チェック項目 | 何を見るか |
|---|---|
| シークレット漏洩 | ソースコードに直書きされた API キー・パスワード |
| インジェクション | SQL インジェクション、コマンドインジェクション、未検証の入力値 |
| 認証・認可 | 認証チェック漏れ、過剰な権限、JWT の扱い |
| 依存関係 | 既知の脆弱性を持つ古いパッケージ |
| CORS / Headers | 過剰に緩いCORS設定、セキュリティヘッダーの欠如 |

---

### 設計 (Design)

| チェック項目 | 何を見るか |
|---|---|
| 責務の分離 | ビジネスロジックがルーター・ビューに混入していないか |
| 結合度 | モジュール間の直接依存が強すぎないか（変更時の影響範囲） |
| 拡張性 | ハードコードされた値、設定化すべき箇所 |
| API 設計 | 命名の一貫性、バージョニング、エラーレスポンスの統一 |
| データフロー | グローバル状態の乱用、データ所有権の不明確さ |

---

### コスト (Cost)

#### 実行コスト
| チェック項目 | 何を見るか |
|---|---|
| N+1 クエリ | ループ内で毎回 DB クエリを発行していないか |
| 計算量 | O(n²) 以上のアルゴリズムが大規模データに適用されていないか |
| メモリリーク | イベントリスナーの解除漏れ、循環参照、肥大化したキャッシュ |

#### API 利用コスト
| チェック項目 | 何を見るか |
|---|---|
| 無駄な呼び出し | 同じデータを繰り返し取得していないか（キャッシュ欠如） |
| ペイロードサイズ | 必要なフィールドだけを取得しているか（過剰フェッチ） |
| リトライ設計 | エクスポネンシャルバックオフがないとコスト急増のリスク |

#### インフラコスト
| チェック項目 | 何を見るか |
|---|---|
| スケーリング設計 | ステートフルなサービスが水平スケールできるか |
| 不要リソース | 使われていないサービス、過剰プロビジョニング |
| CDN / キャッシュ | 静的アセットにキャッシュヘッダーが付いているか |

## スコアリング基準

スコアは正直に付ける。インフレさせない。

| スコア | 基準 |
|--------|------|
| 9-10 | 本番品質。ほぼ改善不要 |
| 7-8 | 良好。軽微な改善の余地あり |
| 4-6 | 改善が必要。リリース前に対処を推奨 |
| 1-3 | 重大な問題あり。即時対応が必要 |

問題が1つも見つからない軸でも、10点は「完璧」なので滅多に付けない。
